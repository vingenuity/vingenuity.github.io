Site Rework Part 2

Generating Resumes Automatically

As mentioned in my previous post, one of the primary reasons for the site rework was to reduce the amount of repetition and copypasta done across the website. The primary offender with repetition by far in my previous workflow was the resumes. In today’s post, I am going to show my new workflow for producing my resumes that heavily reduces this repetition.

Document Formats

Before discussing the workflows, it’s worth taking a moment to enumerate some of the common document formats, as well as why they are useful for resumes.

  • DOCX
    • DOCX is the modern version of Microsoft’s Word application. (The original format was DOC.)
    • Due to the ubiquity of Microsoft Office software in the workplace, this is by far the most common document format requested when applying for work.
    • Even though it is based in XML, due to Word’s proprietary nature, there aren’t many tools that can work with this format.
  • PDF
    • PDF is Adobe’s semi-proprietary document format.
    • While it’s not quite as popular as DOCX, Adobe’s PDF format is generally accepted by most workplaces.
    • Additionally, since PDF format is semi-open, there are many tools that can read, write, and modify this format.
  • HTML
    • HTML is the format used for rendering content in the browser.
    • Even though there are now web-based document renderers for the previous two document formats, having an HTML version of your resume is useful as a fallback when the other methods are not working correctly.
  • TXT
    • TXT is the most basic of text document formats, with little to no formatting support.
    • Many job submission websites request a TXT version of your resume. This is very rarely viewed by humans; instead, the TXT resume will be parsed for keywords by some variety of HR software.

Ideal Workflow

In an ideal world, my workflow would look something like the diagram below:

In this workflow, content and styling are fully separated from each other, as in proper UI practice. Furthermore, since every document’s content and styling are sourced from the same pair of files, I would only have to edit a single document in order to update everything. Additionally, due to being done via a script, the update process is quick and easy.

Unfortunately, with the feature sets of the software of which I am aware, there is no way to produce a workflow with this ideal structure. However, let’s take a look at my new workflow, which is definitely closer to this ideal than before…

New Workflow

Here is a diagram of the new workflow I am using now:

As the chart shows, the new workflow has significantly in both repetition and automation. When updating content, only one file needs to be manually updated instead of three. Additionally, the entire export/generation process is automatic once content has been generated.

However, there are still definitely some flaws with this new workflow. Two style files must be kept synchronized (DOCX and CSS) when updates are made, which is exactly the same level of work as before. Additionally, the TXT version is not currently in the workflow, which I will discuss more about in a later section.

Now let’s walk through the stages of the workflow and see how each works.

Generation Script

Let’s start the walkthrough of the new workflow by looking at the generation script. Here are the important parts of the script:

    
        
param([string]$markdown_content_path, [string]$output_file_name,
      [string]$style_css_path, [string]$style_docx_path, [string]$yaml_metadata_path,
      [int]$pdf_margins_mm=13)

$current_date_formatted = Get-Date -UFormat "%Y-%m-%d"
$pandoc_args_standard = @($markdown_content_path, $yaml_metadata_path, "--standalone", "--metadata", "date=`"$current_date_formatted`"")

$pandoc_arguments_docx = @("-o", "$output_file_name.docx", "--from", "markdown+bracketed_spans", "--to", "docx+styles");
$pandoc_arguments_docx += $pandoc_args_standard;
if($style_docx_path)
{
    $pandoc_arguments_docx += "--reference-doc=$style_docx_path";
}
Call-WithRedirection pandoc $pandoc_arguments_docx

$html_output_file_name = "$output_file_name.html"
$pandoc_arguments_html = @("-o", $html_output_file_name, "--section-divs");
if($style_css_path)
{
    $pandoc_arguments_html += "--css=$style_css_path";
}
$pandoc_arguments_html += $pandoc_args_standard;
Call-WithRedirection pandoc $pandoc_arguments_html

# wkhtmltopdf is not marked as a DPI-aware application in Windows, so by OS convention, it always sees the DPI setting as 96.
$wk_zoom_offset = Calculate-ZoomOffsetForDPI 96;
$wk_arguments = @("--quiet", "--zoom", $wk_zoom_offset);
$wk_arguments += @("--margin-top", $pdf_margins_mm, "--margin-bottom", $pdf_margins_mm, "--margin-left", $pdf_margins_mm, "--margin-right", $pdf_margins_mm);
$wk_arguments += @($html_output_file_name, "$output_file_name.pdf")
Call-WithRedirection wkhtmltopdf $wk_arguments
